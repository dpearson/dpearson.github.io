Software duct tape

Everyone who has written even a few lines of code has heard the mantra: "ship early, ship often." It's not entirely bad advice; sometimes, you do indeed have to make some compromises to get software out the door.

There is, however, an unfortunate tendency when rushing to finish something, especially with a hard deadline looming. I've taken to calling it (after a physical analogue) "duct tape." In it nutshell, duct taped software is anything you're afraid to touch, lest something break. It defies refactoring attempts, and it has tremendous staying power.

My cringeworthy example? NHLScoreBot. It was completely rewritten this spring to finally fix an awful mistake: initially, it was a complete fork of the codebase for the football versions.

That worked great for a while. There was a lot of shared code around summary generation, and I had to quickly ship to present in a competition. Everything mostly worked, too; I had spent a bit of time chasing down all of the football-specific code. As long as I didn't breathe too loudly, it churned out decent summaries.

Eventually, though, I had had enough. I spent a couple of weeks worth of free time extracting all of that shared summary-generation code into its own library. There were still sport-specific changes, but those are now miniscule. I got a lot of functionality for free, without difficult merges every time something minor changed. In short, it's what I probably should have done initially.

But, looking back, it's hard to fault the duct tape too much. The horribly-rigged mashup of code specific to two completely different sports allowed me to ship quickly and bought me time for a difficult refactor. That refactor, too, was probably inevitable, and this one messy project taught me that lesson.

Hopefully, it's a lesson that I won't have to relearn.
