Software duct tape

Everyone who has written even a few lines of code has heard the mantra: "ship early, ship often." It's not entirely bad advice; sometimes, you do indeed have to make some compromises to get software out the door.

There is, however, an unfortunate tendency when rushing to finish something, especially with a hard deadline looming. I've taken to calling it (after a physical analogue) "duct tape." In it nutshell, duct taped software is anything you're afraid to touch, lest something break. It defies refactoring attempts, and it has tremendous staying power.

My cringeworthy example? NHLScoreBot. It was completely rewritten this spring to finally fix an awful mistake: initially, it was a complete fork of the codebase for the football versions.

That worked great for a while. There was a lot of shared code around summary generation, and I had to quickly ship to present in a competition. Everything mostly worked, too; I had spent a bit of time chasing down all of the football-specific code. As long as I didn't breathe too loudly, it churned out decent summaries.

Eventually, though, I had had enough. I spent a couple of weeks worth of free time extracting all of that shared summary-generation code into its own library. There were still sport-specific changes, but those are now miniscule. I got a lot of functionality for free, without difficult merges every time something minor changed. In short, it's what I probably should have done initially.

But, looking back, it's hard to fault the duct tape too much. The horribly-rigged mashup of code specific to two completely different sports allowed me to ship quickly and bought me time for a difficult refactor. That refactor, too, was probably inevitable, and this one messy project taught me that lesson.

Hopefully, it's a lesson that I won't have to relearn.

Experimenting with a different way to learn linear algebra

I should begin with a disclaimer: this is an experiment, but, last week, I started working on a new library, linalgebra, designed as a teaching aide. The algorithms are, almost certainly, uniformly non-optimal. That's not the point, though; I'm curious as to whether taking the time to implement concepts in code will help me learn.

Linear algebra is uniquely suited to this task in ways that other fields, like Calculus, aren't. For one, symbolic math is a pain, even with existing libraries, while linear algebra can be purely numeric and built upon standard data types. Meanwhile, it is imperative that any engineer has a firm grasp on at least basic concepts.

As for the library, it's all written in CoffeeScript, my current favorite language, although, in retrospect (and if I didn't have so many other projects), a project like this one could have been a great opportunity to also learn a new language.

So far, it seems to be helping at least a little, although I now feel like everything math-related I'm analyzing through the prism of Computer Science (for the record, that's not necessarily a bad thing). Still, the jury remains out on whether this project will make any difference for me, and I won't really know that until the day of the final.

At any rate, the code is on GitHub. If it looks like I've abandoned it, please bug me about it on Twitter.

NHLScoreBot, Year II

TL;DR: Hockey's back. Follow NHLScoreBot in English or French for NHL scores and game summaries on Twitter.

With the NHL lockout finally resolved, it's time for another year of hockey. There's a few new features:

  1. Locations. Wanted to know who the home team was? Now, your Twitter client will show a nice map on the summary's detail view.

  2. French support. This feature is something I added last year to NFLScoreBot, and the backend update to NHLScoreBot brought it into the fold.

I've also open sourced some more code. The arena location data I'm using is available on Github (and you can grab it with a standard HTTP request).

My code for using the identi.ca API with node.js is also on Github and NPM.
